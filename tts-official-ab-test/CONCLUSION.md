# Xiaomi MiMo TTS 官方控制 A/B 结论

测试时间：2026-04-29
模型：`mimo-v2.5-tts`
音色：`茉莉`
测试维度：
- `plain`
- `style_only`
- `tag_only`
- `style_plus_tag`

## 客观结果

平均值：

- `plain` → 3.74s / 27.1KB
- `tag_only` → 4.33s / 29.0KB
- `style_plus_tag` → 5.44s / 36.5KB
- `style_only` → 5.81s / 38.9KB

说明：

1. 只要加入 `role:user` 的自然语言风格提示，音频整体时长就明显变长。
2. `tag_only` 比 `plain` 更克制，时长只小幅增加。
3. `style_only` 是最强干预，最容易把表达拉慢、拉满。
4. `style_plus_tag` 仍然明显强于 `tag_only`，但通常略弱于 `style_only`。

## 对赛博女友场景的建议

### 默认策略
优先使用：**`tag_only`**

适用：
- 日常聊天
- 轻微情绪变化
- 不希望声音过演
- 希望保持自然、贴近日常说话

### 实验链路可保留的比较对象
`style_only` 与 `style_plus_tag` 仍可保留用于模型行为测试，帮助理解 Xiaomi 官方能力边界。

但它们**不再属于正式链路候选**。

### 避免策略
避免默认使用：**长风格提示 + 多标签叠加**

原因：
- 容易把语气拖慢
- 容易产生“演出来”的感觉
- 对伴侣型持续对话来说不够日常

## 最终规则

1. **默认只给 `taggedTtsText`，标签数量尽量少**
   - 开头 1 组整体标签即可
   - 中途只有在明显转折处再补 1 个小标签

2. **`naturalStylePrompt` 只在确实需要时启用**
   - 启用时控制在 1 句，最多 2 句
   - 只描述：语气 / 节奏 / 情绪强弱
   - 不写角色背景、不写场景剧本、不写导演说明

3. **高频聊天默认不要上双控制**
   - 当前正式链路完全禁用双控制
   - 双控制只保留给实验脚本，不进正式会话链路

4. **如果文本本身已经带足够明确的标签，优先不再补 style prompt**

5. **目标不是“表现力最大”，而是“像她自然会这么说”**

## 后续代码策略

建议在运行时增加一个简单门控：

- 正式链路：只送 `taggedTtsText`
- `naturalStylePrompt` 在正式链路清空
- 禁止长 director-mode 风格提示进入正式链路
