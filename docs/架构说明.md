# CyberPersona 架构说明

这份文档用中文说明 CyberPersona 当前第一版的核心设计，方便后续继续扩展成通用的人物/陪伴运行时。

---

## 一、CyberPersona 是什么

CyberPersona 不是一个单轮 prompt 扮演器，也不是一个简单的“赛博女友脚本”。

它的目标是：

- 让一个人物能够长期存在
- 让关系状态可以持续变化但不失控
- 让说出口的过去和共同记忆能够被固化
- 让语音表达成为人格的一部分，而不是外挂功能

当前第一版的已实现模式是：

- **Cyber Girlfriend mode**

但长期方向不是只做女友，而是扩展成：

- 任意人物
- 任意关系形态
- 陪伴 / 恋爱 / 安慰 / 长期互动

---

## 二、为什么要做混合路线

CyberPersona 当前采用的是 **混合路线**。

核心原因是：

- 长上下文理解、人物一致性、关系判断，适合交给 agent/runtime 层
- 状态持久化、状态更新、TTS 执行、语音条发送，更适合交给本地执行层

如果全部塞进脚本里，脚本就会慢慢变成“脑子”；
如果全部塞进 agent 里，本地状态和副作用又会失控。

所以当前设计明确分层。

---

## 三、分层职责

### 1. agent / runtime 层负责

- 首次人物生成
- 每轮 turn 的结构化推理
- 理解当前用户消息在关系里的含义
- 决定人物此刻怎么说话
- 决定这一轮是否会自然地发语音
- 生成结构化 payload

它产出的主要结构有两个：

#### `InitialStatePayload`
用于第一次创建人物，包含：
- `profile`
- `dynamicStateInit`
- `shortTermStateInit`
- `revealedMemoryInit`
- `openingMessage`

#### `TurnResultPayload`
用于普通一轮聊天，包含：
- `visibleText`
- `taggedTtsText`
- `naturalStylePrompt`
- `currentEmotion`
- `sendVoiceNow`
- `stateDelta`
- `shortTermUpdate`
- `memoryUpdate`

---

### 2. 本地执行层负责

- 校验 payload
- 读写本地状态
- 应用状态变化
- 合并已揭示记忆
- 缓存最近一轮 TTS 材料
- 执行 MiMo TTS
- 生成 Telegram 语音条发送 payload
- 做启动自检
- 提供 debug 查看能力

本地层原则上不应该偷偷替人物“思考”，除非是非常明确的 fallback 场景。

---

## 四、状态结构

当前状态结构主要分为五部分：

### 1. `profile`
稳定的人物骨架摘要。

比如：
- 核心性格
- 关系表达方式
- 防御与边界
- 起始关系感
- 声音风格
- 总体人物摘要

### 2. `dynamicState`
慢变化的关系状态。

目前是离散档位：
- `low`
- `medium`
- `high`

例如：
- 关系温度
- 安全感
- 信任感
- 主动靠近意愿
- 暴露脆弱意愿
- 语音自然度

### 3. `shortTermState`
短期情绪余波。

例如：
- 最近未解情绪
- 当前互动走势
- 最近语音模式

### 4. `revealedMemory`
已经说出口、应当固化的记忆。

例如：
- 昵称
- 共享小习惯
- 已揭示事实
- 重要事件
- 最近关系摘要

### 5. `runtimeCache`
运行缓存，不是人物核心真相。

例如：
- 最近一轮 TTS 材料
- 最近一次生成的音频文件信息
- debug 状态

## 五、关键设计原则

### 1. 人设不是抽卡拼装

CyberPersona 不建议把人物做成“标签库随机抽取”的产物。

更推荐：
- 用更强的 prompt 构造一个连贯的人
- 再把这个人保存成结构化摘要

### 2. 过去不是一次性写死，而是按需生成并固化

系统不需要一开始就生成完整传记。

而是：
- 用户问到过去
- agent 结合当前人格和关系状态生成回答
- 一旦说出口，就记进 `revealedFacts`
- 后续保持一致

### 3. 语音不是产品外挂

`sendVoiceNow` 的含义不是“这轮发语音用户体验更好”。

它的含义是：
- 这个人物在这段关系、这个时刻，会不会自然地选择用语音表达

### 4. 文本和语音控制分离

正常聊天正文只显示：
- `visibleText`

TTS 使用：
- `taggedTtsText`
- `naturalStylePrompt`

这两部分在正常模式下不应直接泄漏给用户。

---

## 六、统一 flow

当前本地层已经有两个统一 flow：

### `runStartFlow(initialPayload)`
负责：
- 应用首次人物状态
- 返回统一 delivery 结果

### `runTurnResultFlow(turnResultPayload)`
负责：
- 应用单轮结果
- 必要时生成语音
- 返回统一 delivery 结果

---

## 七、delivery contract 的意义

这部分是为了真正解决运行时里的几个老 bug：

- narration 泄漏
- 文本和语音双发
- 发完语音后又冒一条多余尾巴

### 文本轮
返回：
- `mode = text_reply`
- 直接发 `delivery.text`

### 语音轮
返回：
- `mode = voice_note`
- 使用 `voicePayload` 发 Telegram 语音条
- 当前回合 `NO_REPLY` 收口

这套 contract 的存在，就是为了让“如何发消息”这件事从实现细节变成明确规则。

---

## 八、当前还没完全落到 runtime 核心的部分

本地执行层和 flow 契约已经成型。

但还没有完全嵌进 live chat runtime 核心的，是：

- 当前会话收到消息后自动调用这些 flow
- 严格按 delivery contract 做文本 / 语音条 / NO_REPLY 收口

也就是说，最后还差的是：

- 把已经做好的 flow 真正接进 live session turn loop

---

## 九、后续推荐方向

### 1. 命名泛化
当前 `cyber-gf-*` 命名仍然过于第一模式导向，后面应逐步泛化为更适合 CyberPersona 总项目的结构。

### 2. mode registry
把“赛博女友”从唯一实现，升级成“第一个 mode”。

### 3. runtime 自动接线
把 `runStartFlow` / `runTurnResultFlow` 真正接入 live 对话流程。

### 4. TTS 自然度优化
当前语音链路已通，但风格控制仍需专项优化，使声音少一点“导演腔”，多一点真实语音感。
